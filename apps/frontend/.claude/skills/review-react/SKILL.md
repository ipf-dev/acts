## 코드 리뷰 절차

$ARGUMENTS 를 대상으로 코드 리뷰를 수행합니다.

대상 파일을 먼저 읽은 뒤, 아래 체크리스트를 순서대로 검토합니다. 문제가 없는 항목은 생략하고, 발견된 문제만 구체적인 위치(파일명:줄번호)와 함께 보고합니다. 각 문제에는 **현재 코드**, **문제점**, **개선안** 을 포함합니다.

---

### 체크리스트

#### 1. TypeScript 타입 안정성

- [ ] `any` 사용 여부
- [ ] `error: unknown` 없이 에러를 바로 사용하는지
- [ ] 함수 반환 타입 명시 여부 (`: React.JSX.Element`, `: void`, `: Promise<void>` 등)
- [ ] `import type` 사용해야 할 타입을 일반 `import` 로 가져오는지
- [ ] 불필요한 타입 단언(`as`) 사용 여부
- [ ] null/undefined 처리 누락 여부

#### 2. 프로젝트 컨벤션

- [ ] 파일명이 kebab-case 인지
- [ ] Container/Presenter 역할 혼재 여부 (Container에 복잡한 JSX, Presenter에 API 호출)
- [ ] interface 이름에 `View`, `Input`, `Props` 등 접미사 사용 여부
- [ ] 이벤트 핸들러 함수명 `handle` 접두사, props 콜백 `on` 접두사 준수 여부
- [ ] Props 알파벳 순 정렬 여부
- [ ] `cn()` 없이 className 직접 조합하는지

#### 3. React / Hook 정확성

- [ ] `useEffect` 의존성 배열 누락 또는 과잉
- [ ] 비동기 useEffect 에서 `isActive` 클린업 플래그 사용 여부
- [ ] `void` 키워드 없이 `useEffect` 내 async 함수 직접 호출 (`void load()` 형태 사용해야 함)
- [ ] 상태 업데이트를 직접 값으로 하는지 vs 함수형으로 하는지 (이전 상태에 의존할 때는 함수형 필수)
- [ ] 불필요한 `useCallback` / `useMemo` 사용 또는 반대로 누락
- [ ] 컴포넌트 내부에서 객체/배열 리터럴을 props로 직접 전달해 매 렌더링마다 새 참조 생성하는지

#### 4. 상태 관리

- [ ] 하나의 `useState` 로 묶어야 할 관련 상태를 개별 `useState` 로 분산하는지
- [ ] `setState((currentState) => ...)` 함수형 업데이트 사용 여부 (비동기 핸들러 내부)
- [ ] 파생 가능한 값을 별도 상태로 관리하는지 (예: filtered list를 state로 저장)

#### 5. 성능

- [ ] 대용량 리스트를 key 없이 렌더링하거나 index를 key로 사용하는지
- [ ] 불필요한 전체 리스트 재렌더링을 유발하는 구조
- [ ] 컴포넌트 내부에서 API 인스턴스를 매번 생성하는지 (컴포넌트 외부에서 한 번만 생성해야 함)

#### 6. 접근성 (A11y)

- [ ] 인터랙티브 요소(`div`, `span`)에 `onClick`만 있고 `role`, `tabIndex`, `onKeyDown` 누락
- [ ] 이미지에 `alt` 속성 누락
- [ ] form 요소에 연결된 `label` 누락

#### 7. 보안

- [ ] `dangerouslySetInnerHTML` 사용 여부
- [ ] 사용자 입력을 그대로 URL 등에 삽입하는지

### 리뷰 결과 형식

```
## 리뷰 결과: <파일명>

### 🔴 심각 (즉시 수정 필요)
- **파일명:줄번호** — 문제 설명
  - 현재: `코드`
  - 개선: `코드`

### 🟡 경고 (수정 권장)
- ...

### 🔵 제안 (선택적 개선)
- ...

### ✅ 양호한 부분
- 잘 작성된 부분 간략히 언급
```

문제가 전혀 없으면 "리뷰 결과: 특이사항 없음" 으로 보고합니다.
